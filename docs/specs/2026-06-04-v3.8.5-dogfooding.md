**Data:** 2026-06-04
**Sprint:** v3.8.5
**Origem:** Sessao de code review + auto-analise com Mavis (Nando + IA)
**Principio:** usar o DEVORQ para melhorar o DEVORQ

---

## Contexto

O DEVORQ esta na v3.8.4, score de maturidade 8.6/10 (CODE_REVIEW_MATURITY_REPORT de 2026-05-22). O ultimo sprint fechou 4 stories de code review (whitelist SSH, escopo `release`, gates 7->10, remocao do `self-patch`). Mas a sessao de hoje revelou tres dividas tecnicas que precisam ser atacadas em sprint dedicada, sem misturar com features novas:

1. **Cobertura E2E estagnada desde v3.6.0** (2026-05-12, ~35% de sucesso reportado, nunca mais rodou)
2. **`bin/devorq` com 1.503 LOC** - modularizacao do router + dispatchers
3. **`lib/lessons.sh` com 995 LOC** - separacao de CRUD/search/sync/validate
4. **Drift de versao recorrente** - VERSION, bin/devorq, prd.json dessincronizam a cada release
5. **Working dir sujo do sprint v3.8.4** - 5 licoes untracked + PRD orfao

Esta sprint e o **dogfooding real**: o DEVORQ se aplica a si mesmo com a mesma disciplina que ele prescreve.

## Stories

### Story 1 - E2E revival (PRIORIDADE 1)

**Objetivo:** A suite E2E em `e2e-tests/` (Playwright + bash) parou na v3.6.0. Rever, consertar o que quebrou, criar baseline CI com gate obrigatorio.

**Escopo:**
- `e2e-tests/tests/*.ts` - rodar localmente, listar falhas, consertar
- `e2e-tests/RESULTADOS_TESTES.md` - atualizar com data 2026-06-04 e nova % de sucesso
- `scripts/ci-test.sh` - adicionar step de e2e (atualmente so tem unit + security)
- `lib/gates.sh` - adicionar GATE-E2E (nao-bloqueante no dev, bloqueante em CI)
- `.github/workflows/` - criar/atualizar workflow com step de e2e

**Fora do escopo:**
- Adicionar novos testes E2E alem de validar os existentes
- Migrar para outro framework (permanece Playwright)
- Reescrever em outro idioma

**Acceptance criteria:**
- Todos os testes em `e2e-tests/tests/*.ts` rodam sem erro de infra
- Minimo 80% dos testes passam (baseline revisado)
- `RESULTADOS_TESTES.md` reflete a % real pos-fix
- `scripts/ci-test.sh` executa e2e como step
- `devorq verify` reporta status de e2e

### Story 2 - Refactor `bin/devorq` (PRIORIDADE 1)

**Objetivo:** Quebrar `bin/devorq` (1.503 LOC) em router + subcommand dispatchers, cada arquivo <500 LOC.

**Diagnostico atual:**
- `bin/devorq:1-50` - header + bootstrap + version
- `bin/devorq:50-300` - router (parse de comando principal + dispatch)
- `bin/devorq:300-1503` - definicao inline de todos os subcomandos

**Escopo:**
- Mover cada grupo de subcomandos para `lib/dispatchers/<comando>.sh`
- `bin/devorq` vira apenas: header + versao + router + source dinamico dos dispatchers
- Manter 100% da API publica (mesmos comandos, mesmos argumentos, mesmas saidas)
- Manter `devorq::cmd_*` naming convention
- Cada dispatcher com header proprio + responsabilidade unica

**Fora do escopo:**
- Mudar a interface de qualquer subcomando
- Refatorar `lib/*.sh` (apenas `bin/devorq` e `lib/dispatchers/*`)
- Adicionar features novas

**Acceptance criteria:**
- `bin/devorq` < 200 LOC (apenas router + bootstrap)
- `lib/dispatchers/*.sh` com <500 LOC cada
- `devorq --help` produz saida identica a baseline (diff zero)
- `devorq version` retorna 3.8.5
- Suite de testes de seguranca: 11/11 passam
- `bash -n bin/devorq lib/dispatchers/*.sh` retorna 0
- `shellcheck bin/devorq lib/dispatchers/*.sh` retorna 0 errors

### Story 3 - Refactor `lib/lessons.sh` (PRIORIDADE 1)

**Objetivo:** Separar `lib/lessons.sh` (995 LOC) em modulos por responsabilidade.

**Diagnostico atual:**
- `lib/lessons.sh:1-300` - capture + CRUD (create, read, update, delete)
- `lib/lessons.sh:300-600` - search + validate
- `lib/lessons.sh:600-995` - sync (VPS) + compile + migrate

**Escopo:**
- `lib/lessons/crud.sh` - capture, list, get, update, delete
- `lib/lessons/search.sh` - search (com sanitizacao), validate, approve
- `lib/lessons/sync.sh` - sync_vps, compile, migrate
- `lib/lessons.sh` vira agregador que source os 3 modulos
- Manter 100% das funcoes `lessons::*` acessiveis

**Fora do escopo:**
- Mudar a assinatura de qualquer funcao `lessons::*`
- Adicionar novos comandos
- Reescrever logica (apenas mover)

**Acceptance criteria:**
- `lib/lessons.sh` < 100 LOC (apenas source + exports)
- `lib/lessons/{crud,search,sync}.sh` com <400 LOC cada
- Suite de testes de seguranca: 11/11 passam (F-06 grep injection especialmente)
- `bash -n lib/lessons.sh lib/lessons/*.sh` retorna 0
- `shellcheck lib/lessons.sh lib/lessons/*.sh` retorna 0 errors

### Story 4 - `scripts/sync-version.sh` (PRIORIDADE 2)

**Objetivo:** Eliminar drift de versao. Script que varre todos os pontos de versao do projeto e reporta inconsistencias.

**Escopo:**
- `scripts/sync-version.sh` - varre: `VERSION`, `bin/devorq` (header + readonly + help), `lib/*.sh` headers, `CHANGELOG.md` (header de versao atual), `README.md` (badge)
- Detecta drift: `VERSION=3.8.5` mas `bin/devorq` diz `3.8.4`? Reporta.
- Modo `--check` (CI gate, exit != 0 se drift) e modo `--fix` (atualiza tudo para o valor de VERSION)
- Integracao com `devorq verify` (passo opcional)
- Documentacao em `EXTRAS.md`

**Fora do escopo:**
- Auto-bump de versao (continua manual via `devorq commit` com escopo `release`)
- Sincronizacao com tag git
- Migrations de changelog

**Acceptance criteria:**
- `./scripts/sync-version.sh --check` detecta drift entre `VERSION=3.8.4` e `bin/devorq=3.8.3` (cenario real desta sessao)
- `./scripts/sync-version.sh --fix` atualiza todos os pontos para o valor de VERSION
- Integrado como step em `scripts/ci-test.sh`
- Documentado em `EXTRAS.md`

## Riscos e Mitigacoes

| Risco | Severidade | Mitigacao |
|-------|-----------|-----------|
| Refactor de `bin/devorq` quebra dispatcher | Alta | Testar comando por comando apos mover, manter baseline de `devorq --help` |

| Time estimado estoura | Baixa | Stories 2 e 3 podem ser divididas em sub-stories se ficarem > 2h cada |
